# PRD: 점심 메뉴 추천 서비스 (뭐먹지)

## Problem Statement

점심 메뉴를 고르기 힘들다. 지도 앱이나 배달 앱에서 음식점을 검색하면 수십 개의 선택지가 리스트로 쏟아지는데, 선택지가 많으면 오히려 선택 장애가 생긴다. 텍스트 위주의 리스트를 스크롤하며 비교하는 방식은 결정을 더 어렵게 만든다.

음식 사진을 한 장씩 크게 보여주고, 각각에 대해 "좋아요" 또는 "패스"라는 구체적인 yes/no 판단만 하게 하면 선택 장애를 극복할 수 있다. 시각 정보를 극대화하고, 의사결정을 이진 선택으로 단순화하는 것이 핵심이다.

## Solution

JSON 기반의 정적 음식점 데이터를 사용한 점심 메뉴 추천 서비스를 만든다. 지역 선택 UI는 존재하지만 현재는 "교대" 지역으로 고정되어 동작한다. 교대 지역의 음식점 리스트를 랜덤으로 셔플하여 틴더 스타일 Swipe 카드로 보여준다. 오른쪽 스와이프(좋아요) / 왼쪽 스와이프(패스)를 반복하다 최종 1곳이 남으면 그 식당을 추천 결과로 보여준다. 로그인 없이 바로 사용 가능하며, 모바일 우선 디자인(데스크톱은 모바일 뷰 max-width 가운데 정렬)으로 구현한다.

## User Stories

1. 사용자로서, 앱에 접속하면 로그인 없이 바로 지역 선택 화면을 볼 수 있다. 빠르게 서비스를 이용하기 위해서다.
2. 사용자로서, 지역 선택 UI를 볼 수 있다. 현재는 "교대"로 고정되어 있지만, 향후 다른 지역이 추가될 수 있음을 인지하기 위해서다.
3. 사용자로서, 지역을 선택하면 해당 지역의 음식점이 랜덤 순서로 카드 형태로 표시된다. 편향 없이 다양한 선택지를 보기 위해서다.
4. 사용자로서, 카드에서 음식점 이름, 카테고리(한식/중식/일식 등), 대표 이미지를 확인할 수 있다. 스와이프 전에 기본 정보를 파악하기 위해서다.
5. 사용자로서, 카드를 오른쪽으로 스와이프하면 "좋아요"로 표시된다. 마음에 드는 식당을 남기기 위해서다.
6. 사용자로서, 카드를 왼쪽으로 스와이프하면 "패스"로 해당 카드가 사라진다. 관심 없는 식당을 빠르게 넘기기 위해서다.
7. 사용자로서, 좋아요한 식당이 1곳만 남으면 자동으로 최종 결과 화면으로 이동한다. 최종 1곳을 확정하기 위해서다.
8. 사용자로서, 최종 결과 화면에서 식당 이름, 카테고리, 이미지를 확인할 수 있다. 최종 선택을 확인하기 위해서다.
9. 사용자로서, 결과 화면에서 "다시 추천받기" 버튼을 눌러 처음부터 다시 시작할 수 있다. 결과가 마음에 안 들 때를 위해서다.
10. 사용자로서, 모든 카드를 왼쪽으로 스와이프(전부 패스)하면 "추천할 식당이 없습니다. 다시 시도해주세요" 메시지를 볼 수 있다. 빈 상태를 명확히 알기 위해서다.
11. 사용자로서, 모바일에서 터치 스와이프로 자연스럽게 카드를 넘길 수 있다. 모바일 사용성을 위해서다.
12. 사용자로서, 데스크톱에서도 드래그 또는 버튼 클릭으로 좋아요/패스를 할 수 있다. 마우스 환경에서도 사용하기 위해서다.
13. 사용자로서, 스와이프 중 남은 카드 수를 확인할 수 있다. 진행 상황을 파악하기 위해서다.

## Implementation Decisions

### 기술 스택
- 기존 PFC Technologies 프로젝트 활용: TanStack Start + React 19 + TanStack Router + Tailwind CSS v4 + Vite
- 아이콘: lucide-react (기존 의존성)
- Swipe 제스처: 터치/마우스 이벤트 직접 구현 (외부 라이브러리 최소화)

### 데이터
- 음식점 데이터는 JSON 파일로 관리한다 (예: `src/data/restaurants.json`)
- 각 음식점 항목: `{ name, category, imageUrl }`
- 현재는 "교대" 지역 데이터만 포함한다
- 지역별로 JSON 파일을 분리하거나, 하나의 파일에서 지역 키로 구분한다

### 라우트 구조
- `/` — 지역 선택 화면 (현재 "교대" 고정)
- `/swipe` — 스와이프 카드 화면
- `/result` — 최종 추천 결과 화면

### 주요 모듈

1. **Location Selector 모듈**: 지역 선택 UI. 현재는 "교대"가 고정 선택되어 있으며, 다른 지역은 선택 불가(비활성) 상태로 표시한다. 향후 지역 추가를 고려한 구조로 만든다.

2. **Restaurant Data 모듈**: JSON 파일에서 선택된 지역의 음식점 목록을 읽어온다. 결과를 랜덤 셔플하여 반환한다.

3. **Swipe Card 모듈**: 틴더 스타일 카드 UI. 터치 드래그 + 마우스 드래그 지원. 오른쪽 스와이프 = 좋아요, 왼쪽 스와이프 = 패스. 스와이프 애니메이션(회전 + 페이드아웃). 하단에 좋아요/패스 버튼도 제공(데스크톱 접근성).

4. **Swipe State Manager**: 전체 카드 목록, 현재 인덱스, 좋아요 목록을 관리하는 상태 로직. 좋아요 목록이 1개가 되면 결과 페이지로 네비게이션. 모든 카드 소진 시 빈 상태 처리.

5. **Result Display 모듈**: 최종 선택된 식당 정보 표시. "다시 추천받기" 버튼.

### 레이아웃
- 모바일 우선: 전체 앱을 `max-w-md mx-auto` 컨테이너로 감싸서 데스크톱에서도 모바일 뷰 유지
- 기존 Header/Footer 컴포넌트는 이 서비스에 맞게 수정하거나 간소화

### Swipe 동작 상세
- 스와이프 임계값: 화면 너비의 30% 이상 드래그 시 확정
- 임계값 미만 드래그 시 원위치로 스냅백
- 드래그 중 카드 회전: 드래그 거리에 비례 (최대 ±15도)
- 좋아요 시 초록색 오버레이, 패스 시 빨간색 오버레이 힌트

## Testing Decisions

- 좋은 테스트란 내부 구현이 아닌 외부 동작(사용자가 보는 결과)을 검증하는 것이다.
- **Swipe State Manager**: 카드 추가/제거, 좋아요 목록 관리, 최종 1개 남았을 때 판정 로직을 단위 테스트한다.
- **Restaurant Data 모듈**: JSON 데이터 로드 + 랜덤 셔플이 올바른 형태를 반환하는지 테스트한다.
- 기존 프로젝트에 vitest + @testing-library/react가 이미 설정되어 있으므로 동일한 패턴을 따른다.

## Out of Scope

- 로그인/회원가입 기능
- 찜 목록(좋아요 히스토리) 저장
- 추천 알고리즘 고도화 (평점순, 거리순, 개인화 등)
- GPS 기반 자동 위치 감지
- 외부 API 연동 (네이버 지도, 카카오맵 등)
- 음식점 상세 페이지
- 리뷰/평점 자체 수집
- 다국어 지원
- PWA / 네이티브 앱
- 교대 이외 지역 데이터

## Further Notes

- 음식점 이미지는 외부 URL(Unsplash 등 무료 이미지 또는 직접 촬영)을 사용한다.
- 지역 추가 시 JSON 파일에 데이터를 추가하고 Location Selector에서 활성화하는 방식으로 확장한다.
- Swipe 제스처는 외부 라이브러리 없이 React의 pointer events로 직접 구현하여 번들 사이즈를 최소화한다.
- 모바일 우선이므로 `prefers-reduced-motion` 미디어 쿼리를 존중하여 애니메이션을 줄이는 옵션을 제공한다.
